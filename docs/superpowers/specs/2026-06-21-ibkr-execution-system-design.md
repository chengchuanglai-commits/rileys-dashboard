# IBKR 完整执行系统 设计 (Execution System)

日期:2026-06-21 · 作者:Riley + Claude

## 北极星

让系统在 Riley 睡觉时(美股开盘=国内凌晨)自动按引擎配置交易、自动止损保护、醒来给一份详尽复盘。**先 paper 跑稳,真钱是后续一个 config 开关**。盈利为中心,服务 [[project-investment-goal]]。

## 背景

今天(2026-06-21)打通了 IBKR paper 执行链路地基(`scripts/ibkr_connect_test/reconcile/execute.py`,commit 2013174):连接 DUQ627583、只读对账(3场景验证)、10笔限价单真提交+独立回读。但暴露 6 大缺口,本系统补齐:
1. **只有买、没有卖/止损**(致命)
2. 无回撤断路器(Riley 定 30% 红线还没落地)
3. 无开盘前自检(今天踩过"只读没关")
4. 无每日自动化
5. 无成交↔模拟盘对账(滑点没记)
6. 容错弱(跨session撤单要 reqGlobalCancel——今天的教训)

## 三层架构(各干一件事,互不污染)

```
① 决策层 (daily batch)   —— 唯一能下单的层
② 保护层 (broker-side)   —— -8% stop order 挂 IBKR,实时兜底,睡着也保护
③ 复盘层 (post-close)    —— 收盘后生成详尽复盘,推飞书+dashboard
```
**铁律:只有决策层能下单;复盘层只读、只报警,绝不交易。** 观察可频繁容错,交易保持低频严格——把今天踩的坑隔离在决策层一处调试。

## 时间线(美东 → 北京)

| 收盘后 | 次日早 | 复盘层 | 生成详尽复盘,醒来看 |

## 决策层细节

**开盘前自检(只检查,失败即中止+报警,带"原因+建议动作"):**
1. 网关在线、连得上端口4002? 2. DU/paper账户?(防误连实盘) 3. 只读模式确实关了? 4. allocation.json 新鲜(今天更新)? 5. 净值读得到、现金够? 6. 有无遗留孤儿挂单? 7. 行情取价通、关键票有价? 8. 距回撤红线多远、是否已破?

**开盘 batch:** 自检过→拉目标(allocation.json)+实际持仓→对账算差额(补差/清掉榜/持平不动)→安全闸(单笔上限/总额上限/断路器)→下**限价单(IBKR延迟价×缓冲,整数股)**→记 orderId→成交后对每个新持仓挂 **-8% 固定 stop**(券商端)→落 `data/exec-log/{date}.json`。

**收盘 batch:** 健康检查→MOM-MA出场判定(持仓收盘<20MA→挂次日开盘卖单)→周再平衡日(掉榜票→挂卖单)→落 exec-log。

**关键技术决策(已拍板):**
- 下单价 = **IBKR 延迟价做限价单**(够准、和执行同源;paper无实时行情用延迟价)。
- 止损 = **固定 stop order**(到-8%触发市价卖,保命优先,极端时要的就是出得掉)。
- **股数:`FRACTIONAL` 开关(当前=整数股,权限开后切小数股)**。
  - **实测结论(2026-06-21,paper账户)**:cashQty→报错10244「不能用现金数量」、小数股数→报错10243「不能通过API下小数股」。**两条路都堵 = 账户"Trade in Fractions"权限未开**(卡在实盘入金墙,进不去Client Portal开)。**不是API用法问题**(两种正确写法都试了)。
  - **当前 `FRACTIONAL=false`:整数股**(向下取整,买不起1股的高价票跳过)。先建系统、paper跑通——三层/自检/止损/断路器/复盘/容错都与股数无关。
  - **`FRACTIONAL=true`:小数股(限价单+cashQty)**——权限开通后切换,$2000本金也能买全10只、保留frac10分散。**只是一个开关,架构不动。**
  - **⚠️ 必须解决的待办(Riley 强调)**:开 "Trade in Fractions" 权限。卡点=实盘入金墙。解法:①取消失败的pending存款②客服ticket清卡住的待办③进Portal勾权限。持续提醒 `fractional-permission-reminder.py`(每周一飞书,自终止)。**真钱前必须解决**,否则 $2000 分散方案不成立。

## 保护层细节

- **单仓 -8% 固定 stop**:开盘成交后立即挂,券商端常驻,脚本断了也在线。
- **组合级回撤断路器(分档,软着陆非硬撞墙)**:决策层每次先算 净值vs历史峰值 回撤。
  - **阈值=可配置常量,标注「待回测校准」**(Riley 正确指出:不该拍脑袋定;现放安全占位)。
  - 占位档(`# TODO 待回测校准`):**回撤>20% 飞书预警 / >25% 只减不增(停开新仓,只允出场) / >28% 全线挂出场转防御**。
  - 后续专门用 `backtest-momentum-sized.py`(含2022熊市)跑不同阈值,用数据定值。**不阻塞主系统。**

## 复盘层细节(Riley 要求:真钱前尽可能详尽,异常顶到最前)

数据源:`exec-log/{date}.json`(决策层落盘) + IBKR实际持仓/成交 + 引擎目标 → 三方对比。
- **🔴 异常区**(有才显示,无则"今日无异常"):下单失败/拒单 · 滑点超阈值(哪只差多少) · 该成交没成交 · 持仓对不上模拟盘 · 逼近回撤红线 · 止损触发 · 连接/网关/数据异常。
- **📊 动作区**:今日下了什么单 · 成交均价 · 持仓变化 · 现金/净值/当日盈亏。
- **📈 状态区**:组合 vs 模拟盘 vs SPY · 距红线多远 · 各腿信念变化。
- 推送飞书一条 + 落 `data/review/{date}.json` 供 dashboard 历史展示。

## 文件结构

```

  client.py      连接+健康检查(连网关/DU校验/只读校验/重连),所有层共用
  reconcile.py   对账(目标vs实际→差额),纯逻辑可离线测(已验证3场景)
  orders.py      下单封装(限价/stop/整数股/安全闸),唯一下单入口

  trade_open.py  开盘batch
  trade_close.py 收盘batch

data/review/{date}.json     每日复盘存档
config:LIVE开关(默认false=paper)、断路器阈值、安全闸上限、名义本金,集中一处
```

## 容错(今天教训内建)

- 连接失败→重试3次→仍失败报警中止(不蒙眼下单)。
- 撤单用 `reqGlobalCancel`(跨session,今天踩的坑)。
- 每笔下单 try/except,单笔失败不拖垮整批,记进异常区。
- 网关每日强制登出→自检会发现→报警重登(paper先手动,后续IBC自动)。

## 运行环境

- **执行层必须在 Riley 的 Mac 本地跑**(launchd):网关在本地,GitHub Actions 云端连不到 → 这是和数据/dashboard(云端)的根本不同。
- Mac 接电源不自动睡眠(Riley 确认可一直开)。收盘batch若Mac睡眠,醒来补跑可容忍(出场用收盘价)。
- 客户端:**IB Gateway**(比TWS轻+稳,组件少=故障面小,业界自动交易标准;手动救急临时开App)。可靠性真正投资在 **IBC(自动重登)+ 脚本容错**,非纠结客户端大小。
- 内存:Gateway ~300-600MB 常驻(现代Mac无感);执行脚本一天醒3次每次几分钟,跑完即退,几乎0占用;无盘中轮询。
